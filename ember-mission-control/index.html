§01

The Big Idea

Ember CRM should not become a traditional human-first CRM with agents bolted on. It should become the durable place where Ember Bots can see, supervise, and improve every customer agent.

Customer agents are not just installed. They are bounded, monitored, reviewed, improved, and managed. The CRM is the system that makes that true.
Table Stakes

Agent Control Plane

Identity, scope, telemetry, work items, evidence, approvals. Important — but every serious player will have this within 18 months. It will not differentiate.

★ Moat

Managed Calibration

Turning real customer-agent interactions into structured findings — then into context, policy, runbook, entitlement, workflow, and product improvements. With evidence, approval, and auditability.

Non-negotiables
  • No default allow
  • No broad customer-agent authority
  • Account-only matches don't authorize scoped writes when both dimensions exist
  • Every meaningful write needs audit
  • Required creates need idempotency keys
  • Mutable transitions need concurrency protection

  • Production tokens / authority changes require explicit approval
§07

Three customer data postures

Build A first. Add B where useful. Use C only when customer lacks a useful CRM and tenant isolation is fully ready.

MODE A

Monitor-only Control Layer

▸ BUILD FIRST · PROVE FIRST

CRM stores health, work, evidence, approvals, QA, context. Customer CRM remains system of record. Lowest risk, highest leverage.

// order: 1 of 3
MODE B

Overlay + External References

CRM stores operational state. References external records by system/id. Tracks freshness, staleness, drift. Useful where customer already has CRM.

// order: 2 of 3
MODE C

System of Record

CRM stores customer operational records directly. Requires strict tenant isolation and explicit approval. Last, not first.

// order: 3 of 3

§10

Design principles

The non-obvious commitments. The ones we hold when pressure mounts.

01

Agents are primary operators. Humans are supervisors and decision-makers, not data-entry clerks.

02

Operational truth, not vanity dashboards. Show what the system actually did.

03

Every action: scope, authority, evidence, audit. Non-negotiable.

04

Approvals are batched and risk-tiered. No constant manual babysitting.

05

Data posture is explicit. Monitor-only, overlay, or system-of-record. Pick deliberately.

06

Build for Blaze first. Generalize second. Don't pre-abstract.

07

Managed calibration is the moat. Everything else is table stakes.

08

No production authority expansion without explicit approval. Ever.

09

No secrets in Git, memory, docs, fixtures, or logs. Anywhere.

10

Beautiful UI is secondary to making the operational loop trustworthy.
